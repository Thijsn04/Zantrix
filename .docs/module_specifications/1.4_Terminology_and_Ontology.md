# Module 1.4: Terminology & Ontology Server

## 1. Introductie
Een medisch dossier is waardeloos zonder gestandaardiseerde, machine-leesbare betekenis. De Terminology & Ontology Server vormt de taalkundige en semantische ruggengraat van het Zantrix EPD. Deze module verzorgt het laden, indexeren en pijlsnel doorzoekbaar maken van wereldwijde en lokale medische ontologieën zoals SNOMED CT, LOINC, ICD-10, en de lokale ziekenhuis (DHD) codesets. De module is gebouwd bovenop HAPI FHIR en gebruikt Elasticsearch voor milliseconde-prestaties, essentieel voor type-ahead functionaliteit in de hele applicatie (bijv. bij het zoeken naar een diagnose of labtest).

## 2. Kernfunctionaliteiten (Enterprise Scope)
* **High-Performance Search:** Sub-milliseconde auto-complete en fuzzy search op honderdduizenden concepten (bijv. typ "hartaanv" en vind "Myocardinfarct (SNOMED: 22298006)").
* **Ontology Import Pipeline (ETL):** Geautomatiseerde achtergrondprocessen voor het inlezen en updaten van grote bestanden (zoals de SNOMED CT RF2 release files, of LOINC CSVs).
* **Cross-Mapping (ConceptMap):** Het vertalen van de ene codeset naar de andere, bijvoorbeeld het mappen van een lokale ziekenhuiscode naar een SNOMED CT concept, of van SNOMED naar ICD-10 ten behoeve van facturatie (DBC).
* **Value Set Subsetting:** Het creëren van specifieke, kleinere lijsten (ValueSets) uit de grote ontologie. Bijvoorbeeld een ValueSet "Toegestane diagnoses voor specialisme Oogheelkunde" om de dropdowns voor oogartsen relevant te houden.
* **Hierarchische Validatie:** Het valideren of een gekozen concept een "kind" (descendant) is van een ander concept (bijv. valideren of "Bacteriële longontsteking" behoort tot de groep "Infectieziekten").
* **Post-coordination Ondersteuning:** Ondersteuning voor het samenvoegen van concepten, hoewel pre-coordination (SNOMED) de standaard is.

## 3. Datamodel (FHIR Resources)
De server implementeert de HL7 FHIR R4 Terminology services. In Zantrix worden deze data niet primair in PostgreSQL, maar in **Elasticsearch** (of OpenSearch) opgeslagen wegens de zoekvereisten.

### 3.1 CodeSystem & Concept
* **Index:** `terminology_concepts`
* **Velden:**
  * `id` (UUID): Interne ID.
  * `system` (String): De URI van het CodeSystem (bijv. `http://snomed.info/sct`).
  * `code` (String): De unieke code (bijv. `22298006`).
  * `display` (String): De voorkeursterm ("Myocardial infarction").
  * `synonyms` (List<String>): Alle bekende synoniemen ("Hartaanval", "MI", "Hartinfarct").
  * `active` (Boolean): Of de code nog geldig is in de huidige release.
  * `parents` / `children` (List<String>): Navigatie-paden voor de hiërarchie.

### 3.2 ValueSet
Mapping naar de FHIR `ValueSet` resource.
* Bevat de metadata en de expliciete lijst of logische regels (bijv. "Alle codes uit SNOMED die vallen onder concept X") om een set te vormen.

### 3.3 ConceptMap
Mapping naar de FHIR `ConceptMap` resource voor vertaaltabellen.

## 4. API Endpoints
Deze endpoints volgen strak de FHIR Terminology Service (CTS2) specificaties.
* **`GET /fhir/ValueSet/$expand?url={valueSetUri}&filter={searchTerm}`:** Breidt een ValueSet uit gebaseerd op een zoekterm. Wordt gebruikt door 90% van de frontend dropdowns.
* **`GET /fhir/CodeSystem/$lookup?system={system}&code={code}`:** Haalt de exacte metadata (display, definitie) op voor één specifieke code.
* **`GET /fhir/ConceptMap/$translate?system={sys}&code={code}&target={targetSys}`:** Vertaalt de code naar een ander doelsysteem.
* **`POST /api/v1/terminology/admin/import`:** Beheerders endpoint voor het triggeren van een nieuwe RF2 upload job.

## 5. Frontend Componenten
De frontend bevat zowel herbruikbare widgets voor de klinische modules als een beheeromgeving.
* **`ConceptTypeahead.tsx`:** Een sterk geoptimaliseerde React component met debounce-logica die in de hele Zantrix applicatie wordt gebruikt (bij Diagnoses, Allergieën, Lab).
* **`TerminologyBrowser.tsx`:** Een speciale beheerderstool (in de module "Applicatiebeheer") waarmee informatici door de SNOMED-hiërarchie kunnen browsen, vergelijkbaar met de SNOMED International browser.
* **`ValueSetEditor.tsx`:** Een interface voor CMIO's (Chief Medical Information Officers) om handmatig favorietenlijstjes (ValueSets) samen te stellen per afdeling.

## 6. Uitputtende User Cases (Enterprise Scope)

### 6.1 Klinische Invoer
* **UC-1.4.01 - Snelzoeken Diagnose:** Een SEH-arts vult een "Problem" in bij de patiënt. Hij typt "app" en krijgt direct (binnen 50ms) "Appendicitis (SNOMED: 18526009)" en "Appel allergie" voorgesteld. De lijst is gefilterd op de `ValueSet` 'Clinical Findings'.
* **UC-1.4.02 - Synoniemen Herkenning:** Een cardioloog zoekt op "hartaanval", een leken-term. Het systeem toont "Myocardinfarct", omdat "hartaanval" als synoniem in de Elasticsearch index is opgeslagen.
* **UC-1.4.03 - Leeftijd/Geslacht Validatie:** Een kinderarts typt "Zwangerschap" in voor een 5-jarig jongetje. De Terminology Server waarschuwt dat deze code botst met de demografische constraint (gender = male, age < 10) gekoppeld aan de SNOMED-code.

### 6.2 Data Mapping & Facturatie
* **UC-1.4.04 - Facturatie Mapping:** Een arts voegt SNOMED-code X toe aan de episodelijst. Op de achtergrond draait de Workflow Engine, pakt code X, gebruikt de `ConceptMap/$translate` API om dit te vertalen naar ICD-10 code Y, welke vervolgens naar de Billing module (6.2) wordt gestuurd om gedeclareerd te worden.

### 6.3 Systeembeheer
* **UC-1.4.05 - Maandelijkse Update (ETL):** Nictiz publiceert de nieuwe Nederlandse SNOMED-editie. De functioneel beheerder uploadt de ZIP via de admin portal. Het systeem verwerkt (in de achtergrond via een asynchrone Spring Batch job) de 400.000 concepten, update de actieve status van vervallen codes (inclusief referentiële integriteitschecks op openstaande afspraken) en herindexeert de Elasticsearch cluster met zero-downtime.

## 7. NEN7510 / Security eisen
* **Integriteit van Data:** Ontologieën mogen nooit door reguliere gebruikers aangepast worden. De `/admin/import` endpoints zijn beveiligd en vereisen specifieke system-admin rollen.
* **Zero-Downtime Indexing:** Bij een update worden de oude indexen behouden tot de nieuwe volledig is opgebouwd en gevalideerd (Blue-Green deployment in Elasticsearch), dit waarborgt de operationele continuïteit (beschikbaarheid) vereist door NEN7510 voor kritische zorgsystemen.
